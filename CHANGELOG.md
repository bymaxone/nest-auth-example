## [Unreleased]

### Changed

- Consume `@bymax-one/nest-auth` from npm (`^1.0.4`) instead of the sibling `link:../nest-auth` checkout in all four `package.json` files (root, `apps/api`, `apps/web`, `packages/_probe`). The lib's `1.0.3` release fixes the OAuth callback DTO to accept Google's standard query parameters (`iss`, `scope`, `authuser`, `prompt`, `hd`) and adds five new e2e suites covering the platform surface, MFA disable, email verification, negative paths, and the full `IAuthHooks` lifecycle. The lib's `1.0.4` release adds the `oauth.successRedirectUrl` option used below.
- Restore the Next.js 16 default bundler (Turbopack) in `apps/web` — removed `--webpack` from the `dev` and `build` scripts. The original workaround was only required while the library was consumed as a symlinked workspace package; npm-resolved subpath exports now work under Turbopack out of the box.
- Drop the `next` / `next/*` path overrides from `apps/web/tsconfig.json` — no longer needed once the library is npm-resolved.
- Rewrite `README.md` as a comprehensive, English, badge-rich overview mirroring the upstream `@bymax-one/nest-auth` README format.
- Refresh `docs/OVERVIEW.md §7`, `docs/GETTING_STARTED.md`, and `docs/TROUBLESHOOTING.md` to reflect the npm-package consumption model.

### Fixed

- Wrap five auth pages (`/auth/accept-invitation`, `/auth/forgot-password`, `/auth/login`, `/auth/reset-password`, `/auth/verify-email`) that call `useSearchParams()` in a `<Suspense>` boundary. `next build` previously failed prerendering these pages with both Turbopack and webpack.
- Add `apps/web/next-env.d.ts` (auto-generated by Next.js) to `.prettierignore` to stop spurious `format:check` failures whenever Next regenerates the file.
- **Workspace switcher (FCM #20) — full rebuild**: the previous implementation wrote a `tenant_id` cookie on selection but every authenticated controller reads the tenant from the JWT, so the "switch" never took effect. Worse, the e2e spec used selectors that did not match the rendered DOM and silently skipped with `'Tenant switcher not present in this deployment'`. The new flow is Slack-style: a new `GET /api/account/workspaces` endpoint returns every tenant the signed-in user's email has an active account in, the dropdown lists each workspace (with the current one marked `✓`), and selecting a different workspace POSTs `/api/auth/logout` and full-page-redirects to `/auth/login?tenantId=<destination-slug>`. The seed now creates `admin@example.dev` as two distinct `User` rows — one in `acme`, one in `globex` — sharing the same email. The Playwright spec runs without skips and verifies the entire re-auth flow.
- **`AppJwtAuthGuard` now passes platform tokens through to `JwtPlatformGuard`** ([apps/api/src/auth/app-jwt-auth.guard.ts](apps/api/src/auth/app-jwt-auth.guard.ts)). The lib-mounted `/api/auth/platform/*` routes (`me`, `logout`, `refresh`, `sessions`) are guarded by `@UseGuards(JwtPlatformGuard)` only, with no `@Public()` or app-specific `@SkipJwtAuth()` decorator. The global dashboard `AppJwtAuthGuard` previously intercepted these routes and rejected the platform JWT (wrong `type` claim) before `JwtPlatformGuard` could run — effectively making the lib's platform endpoints unreachable. The guard now inspects the route's `__guards__` metadata and short-circuits to `true` when the route declares `JwtPlatformGuard` AND the Bearer payload says `type: 'platform'`.

### Added — E2E coverage (audit-driven, FCM gaps closed)

Following an audit against the lib's surface (32 FCM rows + extra endpoints), the following e2e specs were added to close coverage gaps:

- **`apps/api/test/new-session-alert.e2e-spec.ts`** — FCM #15: asserts `sendNewSessionAlert` fires on login, the email lands in Mailpit, the `session.new` audit row is written in parallel, and exactly one alert is sent per login. Also wires `AppAuthHooks.onNewSession` to actually dispatch the email (the lib never calls `IEmailProvider.sendNewSessionAlert` itself).
- **`apps/api/test/resend-verification.e2e-spec.ts`** — `POST /auth/resend-verification`: happy path, anti-enumeration for verified accounts, anti-enumeration for unknown emails, decorator metadata under repeated calls.
- **`apps/api/test/password-reset-resend-otp.e2e-spec.ts`** — `POST /auth/password/resend-otp`: happy path completes a reset with the resent code, anti-enumeration, decorator metadata.
- **`apps/api/test/platform-endpoints.e2e-spec.ts`** — FCM #22: `POST /platform/refresh` (rotation + admin record), `POST /platform/logout` (JTI revocation), `DELETE /platform/sessions` (revoke-all), `POST /platform/mfa/challenge` (invalid temp token rejected).
- **`apps/api/test/audit-hooks.e2e-spec.ts`** — FCM #30: the previously-untested audit slugs (`user.register.attempted`, `user.registered`, `email.verified`, `user.login.attempted`, `user.login.succeeded`, `user.logout`, `mfa.enabled`).
- **`apps/web/e2e/oauth-google-click-through.spec.ts`** — FCM #12 (browser layer): clicks "Continue with Google" on login + register, intercepts the browser request, and asserts the lib generates a well-formed Google authorization URL (`/o/oauth2/v2/auth` with `client_id`, `response_type=code`, `redirect_uri`, `scope`, `state`).
- **Mailpit helpers** (`apps/api/test/helpers/mailpit.ts`, `apps/web/e2e/fixtures/mailpit.ts`) now skip new-session alert emails by default so specs polling for a verify OTP / password-reset link do not pick up the side-effect alert that fires during register/login.

### Added — Google OAuth setup + bug fixes

- **`docs/OAUTH_GOOGLE.md`** — step-by-step walk-through (~10 min) covering Google Cloud Console project, OAuth consent screen, client ID creation, env wiring, verification, production checklist, troubleshooting, and how the wiring fits together. Linked from `docs/GETTING_STARTED.md` and `README.md`.
- **`.env.example`** — OAuth section reworked with explicit step-by-step instructions and pointers to where each value comes from.
- **`apps/web/playwright.config.ts`** — the e2e stack now boots the API with placeholder `OAUTH_GOOGLE_*` env vars and the web with `NEXT_PUBLIC_OAUTH_GOOGLE_ENABLED=true` so the OAuth controller mounts and the button renders for the new Playwright spec. Real Google credentials are not used; the spec aborts at the `accounts.google.com` boundary.

### Fixed — Google OAuth wiring

- **Broken href in [apps/web/app/auth/login/page.tsx](apps/web/app/auth/login/page.tsx) and [apps/web/app/auth/register/page.tsx](apps/web/app/auth/register/page.tsx)** — the "Continue with Google" button pointed to `/api/auth/oauth/google/start` which does not exist; the lib mounts the initiate endpoint at `GET /api/auth/oauth/:provider?tenantId=…`. Both pages now build the correct URL with the active tenant slug embedded in the query string.
- **`tenantId` query param now carries the real CUID, not the slug** ([apps/web/app/auth/login/page.tsx](apps/web/app/auth/login/page.tsx), [apps/web/app/auth/register/page.tsx](apps/web/app/auth/register/page.tsx)). The Google `<a>` now intercepts the click, calls `resolveTenantForLogin(slug)` to get the `Tenant.id` CUID, and only then navigates to `/api/auth/oauth/google?tenantId=<cuid>`. The lib uses `tenantId` verbatim as the FK on `User.tenantId` when creating accounts via OAuth — passing a slug previously surfaced as HTTP 500 from the Prisma FK constraint at callback time. The href fallback still carries the slug for graceful degradation; the onClick handler is the path the browser actually follows.
- **Dead `'default'` slug fallback replaced with `'acme'`** ([apps/web/app/auth/login/page.tsx](apps/web/app/auth/login/page.tsx), [apps/web/app/auth/register/page.tsx](apps/web/app/auth/register/page.tsx), [apps/web/app/auth/forgot-password/page.tsx](apps/web/app/auth/forgot-password/page.tsx)). The seed at [apps/api/prisma/seed.ts](apps/api/prisma/seed.ts) creates `acme` and `globex` — `default` was never a real tenant, so any code path that fell through to it surfaced as a 404 from the slug resolver. Login, forgot-password, and the register-form default value now point to `acme`. Register's `TENANT_OPTIONS` dropdown is expanded from a single placeholder `[default]` to the real seeded `[acme, globex]` pair.
- **Browser OAuth lands on `/dashboard` instead of the JSON payload** ([apps/api/src/auth/auth.config.ts](apps/api/src/auth/auth.config.ts)). The newly-shipped lib option `oauth.successRedirectUrl` is wired to `'/dashboard'` so the OAuth callback issues a `302` after delivering cookies — closing the UX gap that previously left the user on the raw JSON response shape returned by `OAuthController`. Requires `@bymax-one/nest-auth` ≥ `1.0.4`.
