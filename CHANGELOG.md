## [Unreleased]

### Changed

- Consume `@bymax-one/nest-auth` from npm (`^1.0.2`) instead of the sibling `link:../nest-auth` checkout in all four `package.json` files (root, `apps/api`, `apps/web`, `packages/_probe`).
- Restore the Next.js 16 default bundler (Turbopack) in `apps/web` — removed `--webpack` from the `dev` and `build` scripts. The original workaround was only required while the library was consumed as a symlinked workspace package; npm-resolved subpath exports now work under Turbopack out of the box.
- Drop the `next` / `next/*` path overrides from `apps/web/tsconfig.json` — no longer needed once the library is npm-resolved.
- Rewrite `README.md` as a comprehensive, English, badge-rich overview mirroring the upstream `@bymax-one/nest-auth` README format.
- Refresh `docs/OVERVIEW.md §7`, `docs/GETTING_STARTED.md`, and `docs/TROUBLESHOOTING.md` to reflect the npm-package consumption model.

### Fixed

- Wrap five auth pages (`/auth/accept-invitation`, `/auth/forgot-password`, `/auth/login`, `/auth/reset-password`, `/auth/verify-email`) that call `useSearchParams()` in a `<Suspense>` boundary. `next build` previously failed prerendering these pages with both Turbopack and webpack.
- Add `apps/web/next-env.d.ts` (auto-generated by Next.js) to `.prettierignore` to stop spurious `format:check` failures whenever Next regenerates the file.
- **Workspace switcher (FCM #20) — full rebuild**: the previous implementation wrote a `tenant_id` cookie on selection but every authenticated controller reads the tenant from the JWT, so the "switch" never took effect. Worse, the e2e spec used selectors that did not match the rendered DOM and silently skipped with `'Tenant switcher not present in this deployment'`. The new flow is Slack-style: a new `GET /api/account/workspaces` endpoint returns every tenant the signed-in user's email has an active account in, the dropdown lists each workspace (with the current one marked `✓`), and selecting a different workspace POSTs `/api/auth/logout` and full-page-redirects to `/auth/login?tenantId=<destination-slug>`. The seed now creates `admin@example.dev` as two distinct `User` rows — one in `acme`, one in `globex` — sharing the same email. The Playwright spec runs without skips and verifies the entire re-auth flow.
