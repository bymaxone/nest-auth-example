and demonstrable end-to-end in this release:

### Release procedure

For future releases:

1. Land all gating work on `main`.
2. Run `node scripts/audit-library-exports.mjs` — must exit 0.
3. Update `## [Unreleased]` in this file with the new version and date.
4. Update `docs/RELEASES.md` with the new row.
5. Create an annotated tag: `git tag -a vX.Y.Z -m "vX.Y.Z — <one-line summary>"`
6. A human operator runs: `git push origin vX.Y.Z`
7. The `.github/workflows/release.yml` workflow handles Docker images and the GitHub Release.

### Added

- **Silent workspace switch** ([apps/api/src/account/account.controller.ts](apps/api/src/account/account.controller.ts), [apps/api/src/account/account.service.ts](apps/api/src/account/account.service.ts), [apps/web/components/auth/tenant-switcher.tsx](apps/web/components/auth/tenant-switcher.tsx)). Selecting a workspace in the switcher dropdown no longer redirects to `/auth/login` — the API mints a session for the caller's sibling `User` row in the destination tenant without a password re-entry. New endpoint `POST /api/account/switch-workspace` validates ownership server-side (caller's email must have an ACTIVE row in the destination tenant — the same Slack-style email-cluster model the lib's seed pivots on), then delegates token issuance to the lib's new `AuthService.issueTokensForUserId` (v1.0.10+) and cookie delivery to `TokenDeliveryService.deliverAuthResponse`. The frontend calls `useSession().refresh()` + `router.refresh()` after success so every component reading from `useSession()` sees the destination identity on the next render tick. **MFA-enabled destination accounts** trip the lib's `MFA_REQUIRED` guard inside `issueTokensForUserId` — the frontend catches that and falls back to the v1.0.9-style logout + redirect flow so the canonical MFA challenge runs. Other lib errors (`ACCOUNT_SUSPENDED`, `ACCOUNT_BANNED`, 404 stale list) surface as a toast and leave the user on the current workspace. Backed by 6 new unit tests on `AccountService.findSwitchTarget` covering each failure mode, 4 new unit tests on the controller pinning the ownership→issuance→delivery chain, and 2 reframed `TenantSwitcher` tests (happy silent-switch path + MFA fallback path).

### Changed

- Bump `@bymax-one/nest-auth` to `^1.0.10` in all four `package.json` files (root, `apps/api`, `apps/web`, `packages/_probe`). v1.0.10 adds `AuthService.issueTokensForUserId(userId, ip, userAgent)` (password-less token issuance for ownership-proven flows) and promotes `TokenDeliveryService` to a public export so consumer apps can write the lib's canonical auth cookies on their own controllers without replicating the attribute set inline. Both are used by the new silent workspace-switch endpoint described above. Backward-compat: consumers that do not invoke either symbol see zero change.
- Bump `@bymax-one/nest-auth` to `^1.0.9` in all four `package.json` files (root, `apps/api`, `apps/web`, `packages/_probe`). v1.0.9 fixes a silently-broken OAuth + MFA flow when the consuming Nest app uses `app.setGlobalPrefix(...)`: prior to v1.0.9, the `mfa_temp_token` HttpOnly cookie was set by the OAuth callback with `Path` hard-derived from `routePrefix` (`/auth/mfa`), but the real challenge URL is `/api/auth/mfa/challenge` because `main.ts` calls `app.setGlobalPrefix('api')`. The browser refuses to send a cookie whose Path is `/auth/mfa` on a request to `/api/auth/mfa/...` per RFC 6265 prefix-match, so the cookie was always dropped, the `MfaController` received no token, and every challenge attempt surfaced as `MFA_TEMP_TOKEN_INVALID` ("MFA session expired") regardless of whether the user typed the correct TOTP. The lib could not detect this — `setGlobalPrefix` is configured on the Nest app instance after the auth module is built. v1.0.9 introduces `cookies.mfaTempCookiePath?: string` so consumers can pin the exact `Path` attribute the cookie is set with (and cleared with). The example's `auth.config.ts` now sets it to `'/api/auth/mfa'`, matching the global prefix. The new option is also used by `MfaController.clearMfaTempCookie` so the clear path can never diverge from the set path.
- Bump `@bymax-one/nest-auth` to `^1.0.8` in all four `package.json` files (root, `apps/api`, `apps/web`, `packages/_probe`). v1.0.8 is a UX-critical bug fix on top of v1.0.7: the lib's `TokenManagerService.verifyMfaTempToken` previously used an atomic Redis `GETDEL` as its FIRST step, so a single mistyped TOTP digit consumed the JWT — every retry under the same `mfa_temp_token` (sessionStorage for password login, HttpOnly cookie for OAuth) failed with `MFA_TEMP_TOKEN_INVALID` ("MFA session expired. Please sign in again."). v1.0.8 splits verify from consume: `verifyMfaTempToken` now uses a non-destructive `GET`, and a new sibling `consumeMfaTempToken(jti)` performs the `DEL` only AFTER `MfaService.challenge` confirms the code is valid. Wrong codes surface as `MFA_INVALID_CODE` with the JWT still alive for retry inside the 5-minute TTL. The brute-force counter (`bruteForce.recordFailure` keyed on `challenge:${userId}`) still caps total wrong-code attempts per token, so the security model is unchanged. The lib's `MfaController` also refines its cookie-clearing policy: keep the OAuth `mfa_temp_token` cookie on `MFA_INVALID_CODE` / `ACCOUNT_LOCKED` / transient errors (retry possible); clear only on success or `MFA_TEMP_TOKEN_INVALID` (token genuinely dead).
- Consume `@bymax-one/nest-auth` from npm (`^1.0.7`) instead of the sibling `link:../nest-auth` checkout in all four `package.json` files (root, `apps/api`, `apps/web`, `packages/_probe`). The lib's `1.0.3` release fixes the OAuth callback DTO to accept Google's standard query parameters (`iss`, `scope`, `authuser`, `prompt`, `hd`) and adds five new e2e suites covering the platform surface, MFA disable, email verification, negative paths, and the full `IAuthHooks` lifecycle. The lib's `1.0.4` release adds the `oauth.successRedirectUrl` option (wired below). The lib's `1.0.5` release changes the default `cookies.sameSite` from `'strict'` to `'lax'` and exposes it as a configurable option. The lib's `1.0.6` release adds `POST /auth/mfa/recovery-codes` (regenerate) and the entire `/auth/platform/mfa/*` controller surface (setup, verify-enable, disable, regenerate) for platform admins. The lib's `1.0.7` release branches `OAuthService.handleCallback` on `mfaEnabled` (planting a short-lived `mfa_temp_token` HttpOnly cookie + redirecting to `oauth.mfaRedirectUrl`) and adds `oauth.errorRedirectUrl` for graceful provider-error handling — both wired below.
- Restore the Next.js 16 default bundler (Turbopack) in `apps/web` — removed `--webpack` from the `dev` and `build` scripts. The original workaround was only required while the library was consumed as a symlinked workspace package; npm-resolved subpath exports now work under Turbopack out of the box.
- Drop the `next` / `next/*` path overrides from `apps/web/tsconfig.json` — no longer needed once the library is npm-resolved.
- Rewrite `README.md` as a comprehensive, English, badge-rich overview mirroring the upstream `@bymax-one/nest-auth` README format.
- Refresh `docs/OVERVIEW.md §7`, `docs/GETTING_STARTED.md`, and `docs/TROUBLESHOOTING.md` to reflect the npm-package consumption model.

### Removed

- **`packages/_probe` workspace package** ([packages/\_probe/](packages/_probe/)). The probe was a Phase 2 deliverable: a tiny package that imported one symbol from each of the lib's five subpaths (`server`, `shared`, `client`, `react`, `nextjs`) to fail `pnpm typecheck` if the `link:../nest-auth` workspace resolution drifted while the library was still unpublished. The original development plan ([docs/DEVELOPMENT_PLAN.md §Phase 3](docs/DEVELOPMENT_PLAN.md)) scheduled the deletion for the moment `apps/api` started consuming those exports directly — but the cleanup escaped the Phase 3 close-out. Removing it now: the lib is consumed from npm (`^1.0.10`), every subpath is exercised by real application code (`apps/api/src/auth/auth.module.ts`, `apps/web/proxy.ts`, `apps/web/lib/auth-client.ts`, `apps/web/app/providers.tsx`, …) so the typecheck gate already breaks immediately if any subpath export shape changes. Net effect: one less workspace, faster install / typecheck, and the `pnpm-workspace.yaml` glob (`packages/*`) silently picks up the empty directory state. Also removes the dangling "Phase 3 follow-up" entry from the historical block below. `README.md` (architecture tree + test summary) and `docs/OVERVIEW.md §7` updated to drop the probe from the four-package list.

### Fixed

- **WebSocket `reconnect()` no longer spawns zombie sockets** ([apps/web/lib/ws-client.ts](apps/web/lib/ws-client.ts)). The first cut of `reconnect()` (introduced earlier in this changelog block) closed the stale socket and immediately opened a replacement. But `WebSocket.close()` is asynchronous — the stale socket's `onclose` then fired AFTER the synchronous `connect()` had already attached the new socket to the module-level `socket` reference. The stale callback nullified that reference and scheduled a second reconnect via the backoff timer, so every `reconnect()` left 2-3 zombie sockets alive on the server. Visible symptom: every test notification fired the toast 2-3× because the gateway dispatched the same `notification:new` event to all of the user's open sockets, and each was registered as a listener on the same browser tab. Fix: detach `onopen`/`onmessage`/`onclose`/`onerror` from the stale socket **before** calling `close()` so the stale callback cannot race the replacement. The close frame still reaches the server, so `NotificationsGateway.handleDisconnect` still tears down the old socket cleanly. Regression test added in `ws-client.test.ts` pins the "exactly one replacement, no zombies" invariant.
- **WebSocket notifications now reconnect on re-login / tenant switch** ([apps/web/lib/ws-client.ts](apps/web/lib/ws-client.ts), [apps/web/components/notifications/notification-listener.tsx](apps/web/components/notifications/notification-listener.tsx)). The module-level WS singleton would never establish a working connection in the second authenticated session of a browser tab: after a logout the socket was closed and an exponential-backoff timer (up to 30 s between attempts) kept trying to upgrade against the now-unauthenticated `/ws/notifications` endpoint. When the user re-authenticated (different account, different tenant, OAuth-MFA, etc.) the `NotificationListener` re-ran its `useEffect`, but `getWsClient()` only opened a new socket if `socket === null` — and the in-flight reconnect attempts were still pending with no awareness of the cookie change. Two symptoms followed: (1) no `ws:connect` log on the API side, so `POST /api/debug/notify/self` returned `delivered: 0`; (2) eventually a stale-cookie reconnect would land in a window between sessions and authenticate as the wrong identity. Added an explicit `ws.reconnect()` method that cancels the pending backoff timer, resets `attempt` to 0, tears down the current socket, and calls `connect()` synchronously with the browser's current cookies. The `NotificationListener` now calls `reconnect()` whenever `user.id` becomes set — covering re-login (admin@example.dev → Globex), OAuth-MFA completion, and tenant-switch flows in one place. Regression test added.
- **MFA challenge page now revalidates `AuthProvider` before redirecting to /dashboard** ([apps/web/app/auth/mfa-challenge/page.tsx](apps/web/app/auth/mfa-challenge/page.tsx)). Companion to the lib v1.0.9 cookie-path fix: with the OAuth-MFA flow now actually delivering the session cookies, the page successfully reaches /dashboard — but every component reading `useSession()` (Security page conditional `MfaSetupCard`/`MfaDisableCard`, role-gated nav, `NotificationListener` opening its WebSocket) was stuck on `user === null` because the `AuthProvider` was mounted before the MFA challenge ran (when `GET /api/auth/me` legitimately returned 401), and `router.replace('/dashboard')` is a soft navigation that never re-mounts the provider. The page now calls `useSession().refresh()` after the challenge succeeds — forcing the provider to re-fetch `/api/auth/me`, populate `user`, and unlock every downstream component before the soft navigation lands. Both the password-login and OAuth-cookie paths share the same fix because both terminate with the same `router.replace('/dashboard')`.
- **MFA challenge page now lets users retry a wrong code without losing the temp token** ([apps/web/app/auth/mfa-challenge/page.tsx](apps/web/app/auth/mfa-challenge/page.tsx)). Companion to the lib v1.0.8 split-verify/consume fix: on `MFA_INVALID_CODE`, `ACCOUNT_LOCKED`, and transient errors the page now clears the OTP boxes via React Hook Form's `reset({ type, code: '' })` so the user can re-type without manually deleting digits — the lib's temp JWT is kept alive in Redis for retry. The unrecoverable `MFA_TEMP_TOKEN_INVALID` case still purges `sessionStorage` and bounces back to `/auth/login` (no path forward without a fresh login or OAuth drive).
- **`apiFetch` + `platformApiFetch` now propagate the API's `AuthExceptionFilter` envelope code** ([apps/web/lib/auth-client.ts](apps/web/lib/auth-client.ts)). The example's `AuthExceptionFilter` reshapes the lib's nested `{ error: { code, message, details } }` body into a FLAT `{ code, message, statusCode }` envelope, which is the dominant shape on the wire. The previous wrappers only matched the nested envelope and a generic NestJS `{ statusCode, message }` shape — so every `AuthException` (wrong TOTP, invalid credentials, locked account, etc.) ended up with `AuthClientError.code === undefined`, `mapAuthClientError` returned `'UNKNOWN'`, and the user saw the generic "An unexpected error occurred" toast instead of the localized message ("Invalid verification code", "Account locked", …). The wrappers now detect three body shapes in priority order: example's flat envelope → lib's raw envelope → NestJS standard. Tests added on both helpers for every shape including defence-in-depth cases (code present but message missing, no recognizable shape).
- Wrap five auth pages (`/auth/accept-invitation`, `/auth/forgot-password`, `/auth/login`, `/auth/reset-password`, `/auth/verify-email`) that call `useSearchParams()` in a `<Suspense>` boundary. `next build` previously failed prerendering these pages with both Turbopack and webpack.
- Add `apps/web/next-env.d.ts` (auto-generated by Next.js) to `.prettierignore` to stop spurious `format:check` failures whenever Next regenerates the file.
- **Workspace switcher (FCM #20) — full rebuild**: the previous implementation wrote a `tenant_id` cookie on selection but every authenticated controller reads the tenant from the JWT, so the "switch" never took effect. Worse, the e2e spec used selectors that did not match the rendered DOM and silently skipped with `'Tenant switcher not present in this deployment'`. The new flow is Slack-style: a new `GET /api/account/workspaces` endpoint returns every tenant the signed-in user's email has an active account in, the dropdown lists each workspace (with the current one marked `✓`), and selecting a different workspace POSTs `/api/auth/logout` and full-page-redirects to `/auth/login?tenantId=<destination-slug>`. The seed now creates `admin@example.dev` as two distinct `User` rows — one in `acme`, one in `globex` — sharing the same email. The Playwright spec runs without skips and verifies the entire re-auth flow.
- **`AppJwtAuthGuard` now passes platform tokens through to `JwtPlatformGuard`** ([apps/api/src/auth/app-jwt-auth.guard.ts](apps/api/src/auth/app-jwt-auth.guard.ts)). The lib-mounted `/api/auth/platform/*` routes (`me`, `logout`, `refresh`, `sessions`) are guarded by `@UseGuards(JwtPlatformGuard)` only, with no `@Public()` or app-specific `@SkipJwtAuth()` decorator. The global dashboard `AppJwtAuthGuard` previously intercepted these routes and rejected the platform JWT (wrong `type` claim) before `JwtPlatformGuard` could run — effectively making the lib's platform endpoints unreachable. The guard now inspects the route's `__guards__` metadata and short-circuits to `true` when the route declares `JwtPlatformGuard` AND the Bearer payload says `type: 'platform'`.

- **OAuth `Set-Cookie` survives the Google → app redirect**. The lib's `1.0.5` release changes the default `cookies.sameSite` from `'strict'` to `'lax'`. The example does NOT override the option — it relies on the new `'lax'` default and pins it in [apps/api/test/login-and-logout.e2e-spec.ts](apps/api/test/login-and-logout.e2e-spec.ts) so a future lib regression would surface here first.

### Fixed — Security page does not update after MFA toggle

- **`/dashboard/security` now swaps cards immediately after enrolling or disabling MFA** ([apps/web/app/dashboard/security/page.tsx](apps/web/app/dashboard/security/page.tsx)). The page was calling `useRouter().refresh()` — a Next.js Server-Component re-render — but the conditional `mfaEnabled ? <MfaDisableCard/> : <MfaSetupCard/>` is driven entirely by client-side state from the lib's `useSession()`. `router.refresh()` does nothing to that state, so after a successful enroll the user kept seeing the "Set up authenticator" button and any click on it surfaced the "MFA already enabled" toast. Replaced with `useSession().refresh()`, which re-issues `GET /api/auth/me` and updates the `AuthProvider` context in place — the card swap now fires on the very next render tick. Removed the matching `page.goto('/dashboard/security')` workarounds in [apps/web/e2e/mfa-disable.spec.ts](apps/web/e2e/mfa-disable.spec.ts); Playwright's auto-wait already covers the now-instant card transition.

### Fixed — `apiFetch` / `platformApiFetch` now propagate the lib's `AuthException` code

- **Pre-existing bug surfaced on every lib endpoint that returns a structured error** ([apps/web/lib/auth-client.ts](apps/web/lib/auth-client.ts)). The two fetch wrappers (`apiFetch` and `platformApiFetch`) used to throw `new AuthClientError(message, status)` without the third `body` argument. The `mapAuthClientError` helper then saw `error.code === undefined` and returned the sentinel `'UNKNOWN'`, which `translateAuthError('')` resolves to the generic "An unexpected error occurred. Please try again." copy. The first user-visible symptom was a wrong-TOTP submission on the new `/auth/mfa-challenge?source=oauth` flow returning the generic error instead of "Invalid MFA code"; the underlying defect affected every other helper that goes through `apiFetch` (regenerate recovery codes, get MFA status, sessions, audit, etc.). The fix:
  - Both wrappers now detect the lib's `AuthException` envelope shape (`{ error: { code, message, details } }`) AND the standard NestJS shape (`{ statusCode, message, error }`).
  - Either shape is normalised into `AuthErrorResponse` and passed to `AuthClientError` as the third argument, so `error.code` and `error.body.message` round-trip correctly to `mapAuthClientError` / `translateAuthError`.
  - Six new unit tests pin the propagation: AuthException happy-path code, NestJS message fallback, AuthException with omitted message (defence-in-depth), and the same trio on the platform path.

### Fixed — OAuth + MFA users are no longer locked out (lib v1.0.7 wired)

- **OAuth-MFA challenge UX restored** ([apps/api/src/auth/auth.config.ts](apps/api/src/auth/auth.config.ts), [apps/web/app/auth/mfa-challenge/page.tsx](apps/web/app/auth/mfa-challenge/page.tsx), [apps/web/lib/auth-client.ts](apps/web/lib/auth-client.ts)). Before lib v1.0.7, an MFA-enabled user signing in via Google would land on `/dashboard` with session cookies that carried `mfaVerified: false`. The global `MfaRequiredGuard` then rejected every authenticated request with `MFA_REQUIRED`, leaving the user stuck on a `Loading…` Security page with a recurring "Two-factor authentication required" toast on every poll. Lib v1.0.7 fixes the underlying gap (`OAuthService.handleCallback` now branches on `mfaEnabled` and issues an MFA temp token instead of a session); the example wires:
  - `oauth.mfaRedirectUrl: '/auth/mfa-challenge?source=oauth'` — the lib plants the temp token in a short-lived HttpOnly `mfa_temp_token` cookie (Path scoped to `/api/auth/mfa`, 5-minute Max-Age matching the JWT TTL) and 302s here.
  - The `/auth/mfa-challenge` page now detects `?source=oauth` and switches its flow: it skips the `sessionStorage` read (the OAuth path has no token in client-readable storage), and the form's submit handler calls the new `mfaChallengeViaCookie(code)` helper which POSTs to `/api/auth/mfa/challenge` with the code only. `tenantAwareFetch` includes credentials so the browser sends the HttpOnly cookie automatically; the lib's `MfaController.challenge` reads the token from the cookie when the body's `mfaTempToken` is absent and clears the cookie on every outcome. A small "Continuing your Google sign-in — one more step." banner explains the extra step to users coming from OAuth.
  - The page is also refactored to a `<Suspense>`-wrapped inner form because `useSearchParams()` triggers Next 16's CSR bailout (same fix pattern used for the other auth pages).
- **OAuth provider errors land on `/auth/login` with a toast** ([apps/api/src/auth/auth.config.ts](apps/api/src/auth/auth.config.ts), [apps/web/app/auth/login/page.tsx](apps/web/app/auth/login/page.tsx)). Wired `oauth.errorRedirectUrl: '/auth/login'` so any `AuthException` thrown during the OAuth callback (provider error, hook reject, invalid state, etc.) resolves to a 302 to `/auth/login?error=<code>` instead of a raw JSON 500. The login page reads `?error=<code>` in a `useEffect` and fires `translateAuthError` through a `sonner` toast — same translation pipeline the form already used. The query param stays in the URL so a refresh re-surfaces the error until the user acts.
